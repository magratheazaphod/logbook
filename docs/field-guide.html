The Logbook Field Guide

front-end patterns · one page, no build step

The logbook UI is a single index.html: roughly 300 lines of CSS and 650 lines of vanilla JavaScript, no framework, no build step, no webfonts, no icon library. That constraint is what makes it a good textbook — every technique in it is a plain-CSS or plain-DOM idea you can reuse anywhere, including inside React or any other framework later. This guide walks the real code, pattern by pattern, and closes with a ledger of design decisions where an alternative was considered and rejected. The page you're reading is set in the logbook's own design system, so most patterns are demonstrated live in place.

Chapter 1

Design tokens: the whole theme lives in :root

The first 17 lines of the stylesheet are the most important ones in the file. Every color, radius, shadow, and font stack is declared once as a CSS custom property (a "variable") on :root, and the other ~280 lines only ever reference those names: 

:root{
  --paper:#F1E6E4;   /* the page ground */
  --card:#FBFAF9;    /* raised surfaces sit on the paper */
  --ink:#1B1D1E;     /* primary text */
  --muted:#6B6664;   /* secondary text */
  --faint:#A69C9A;   /* tertiary: hints, idle icons */
  --line:#E3D2CE;    /* default hairline border */
  --line-strong:#D2BDB8;
  --accent:#B23A2E;  /* brick red — the one signal colour */
  --accent-ink:#7C2620;
  --accent-soft:#F5DEDA;
  --done:#8A8D8A;
  --radius:10px;
  --shadow:0 1px 2px rgba(20,20,20,.05),0 1px 1px rgba(20,20,20,.03);
  --mono:ui-monospace,"SF Mono",SFMono-Regular,Menlo,Consolas,monospace;
  --sans:-apple-system,BlinkMacSystemFont,"Segoe UI",system-ui,sans-serif;
}

This is the pattern design systems call tokens: named decisions, separated from the components that use them. Three practical payoffs:

  • One knob per decision. Change --accent and every focus ring, status dot, hover state, and timeline dot updates together. There is no find-and-replace across the file, and no drift where two buttons end up slightly different reds.
  • The names encode roles, not colors. --muted, not --gray-600. When you read color:var(--faint) in a rule you know the element's importance without seeing it rendered.
  • Theming becomes trivial. Redefine the same token names in a different scope (a media query, a data-theme attribute) and the whole page re-skins — no component rule changes. This guide's dark mode works exactly that way; the logbook itself deliberately doesn't have one (see the ledger, chapter 9).

Note that --radius, --shadow, and the two font stacks are tokens too. Non-color decisions drift just as easily as colors do.

Alternative discarded: a CSS framework (Tailwind, Bootstrap) or a Sass build would give tokens too — but the project's founding rule is no dependencies, no build step. Custom properties are the standards-native way to get the same discipline for free, and unlike Sass variables they're live at runtime, which chapter 5's resizer exploits.
Chapter 2

Color: a neutral ramp and exactly one signal

Squint at the logbook and it's a warm, quiet field with small red marks on it. That is the entire color strategy, and it has two halves.

The neutral ramp

Text gets three levels of importance — --ink for content, --muted for supporting text (counts, timestamps), and --faint for things you should barely notice until you need them (drag grips, placeholders, idle buttons). Borders get two (--line, --line-strong). Crucially, the neutrals aren't gray: they are rose-tinted grays (#A69C9A, #E3D2CE) that share a hue family with the #F1E6E4 paper. Neutrals mixed toward the page's hue look chosen; pure #999 on a warm ground looks like a default nobody picked.

The accent comes in three strengths

This is the most reusable idea in the file. The brick red isn't one color, it's a trio covering the three jobs a color can have:

  • --accent (B23A2E) — full strength, for small filled shapes: status dots, focus rings, the 2px timeline rail dot.
  • --accent-ink — a darker version that stays readable when the red is used as text on light backgrounds (hover labels, pill text).
  • --accent-soft — a pale wash used as a background behind accent-colored text (pills, callouts, hover states of small buttons).

A common beginner mistake is having only the full-strength color and using it for everything — big red text (too loud, poor contrast) and red-on-red hover states (illegible). One hue, three strengths, three jobs.

Why only one accent?

Because color is a budget. With one signal color, red always means something: "active", "interactive", "look here". The moment a second decorative hue appears, both lose meaning. The logbook spends outside the budget only twice, and both are semantic, not decorative: an amber .warn style for warnings, and a purple pill marking sessions that came from Cowork rather than the CLI — a genuinely different category of data, given a genuinely different hue.

Documented choice: the rose/brick theme is an explicit aesthetic preference recorded in the project history — not a default that happened to ship. The lesson isn't "use brick red"; it's that the ramp-plus-one-signal structure works with any hue you pour into it.
Chapter 3

Typography without webfonts

The logbook loads zero font files. It declares two system font stacks and gets native-feeling, fast, always-available type on every OS: 

--sans:-apple-system,BlinkMacSystemFont,"Segoe UI",system-ui,sans-serif;
--mono:ui-monospace,"SF Mono",SFMono-Regular,Menlo,Consolas,monospace;

The interesting part is the division of labor. The sans face is for prose: titles, task text, summaries. The mono face is for anything data-flavored: timestamps, counts, token totals, pills, branch names, uppercase labels. This gives the page two visual "voices" — reading voice and instrument-panel voice — without a second font download. Mono also aligns digits into columns for free, which is why the session clock times (09:14–11:52) scan so cleanly.

Letter-spacing runs opposite to size

Two rules from the real stylesheet, side by side: 

.brand h1 { font-size:21px; letter-spacing:-.02em; }   /* big → tighten */
.col-head h2 {
  font-size:12px; letter-spacing:.11em;             /* small caps label → open up */
  text-transform:uppercase; color:var(--muted);
}

Large text is tracked slightly tighter (negative letter-spacing) because at display sizes the default gaps look loose. Small uppercase labels are tracked wider, because capitals packed at text spacing become a dense brick. This pairing — tight-big, loose-small-caps — is one of the fastest ways to make an interface look typeset instead of merely rendered.

The scale is narrow and deliberate

Every font size in the file falls between 10.5px and 21px — a factor of two, total. Hierarchy comes mostly from weight, case, color, and family, not from size jumps: a section header is smaller than body text but reads as a header because it's bold, uppercase, letterspaced, and muted. Quiet scales are what make dense utility UIs feel calm.

Chapter 4

Surfaces and depth: paper, card, hairline, whisper

The logbook builds depth from four ingredients, all subtle:

  1. Two surface levels. The page is --paper; anything you can act on (a task row, an input, a button) sits on a lighter --card. Interactive things literally rise off the page.
  2. A hairline border (1px solid var(--line)) in the same warm family, so edges are defined without being drawn attention to.
  3. A whisper of shadow. The shadow token stacks two layers at 5% and 3% opacity: 0 1px 2px rgba(20,20,20,.05), 0 1px 1px rgba(20,20,20,.03). You can't point at it, but remove it and every card goes flat. Layered low-alpha shadows read as ambient light; a single 0 4px 12px rgba(0,0,0,.3) reads as a sticker.
  4. One radius token (10px) for cards and inputs, with smaller one-off radii (8px, 6px, 5px) for nested or tiny elements — inner elements get smaller radii than their containers, which keeps corners looking concentric.

Chapter 5

Layout: flex columns, a sticky rail, and a CSS-variable resizer

The page is two columns — board on the left, daily log on the right — built with flexbox rather than grid, and the reason is instructive. 

.grid      { display:flex; align-items:flex-start; }
.board-col { flex:0 0 var(--board-width, 420px); min-width:0; }
.log-col   { flex:1 1 auto; min-width:0; }

Why flex here: the resizable column

The board column's width is a runtime CSS variable. The drag handle between the columns is ~20 lines of JS that does nothing but update that one custom property and remember it: 

boardCol.style.setProperty("--board-width", w + "px");   // during drag
localStorage.setItem("logbook.boardWidth", Math.round(w)); // on release

Because the stylesheet reads var(--board-width, 420px) — note the fallback value after the comma — the CSS fully defines the default layout and JS only ever nudges one number. This is the cleanest split of responsibilities for dynamic layout: JS owns the value, CSS owns everything the value means. The handle itself is a classic touch-target trick: a 17px-wide invisible strip (easy to grab) drawing a 2px visible line via ::after (visually quiet).

The sticky column with its own scroll rail

The backlog can grow very long while "Today's Focus" — the drop target you drag tasks onto — lives in the right column. If both columns scrolled with the page, scrolling deep into the backlog would push the drop target off-screen. The fix, with the actual comment from the source: 

.board-col{
  /* Own scroll rail: the backlog can grow long, but Today's Focus lives in
     the right column — pinning this pane lets you scroll deep into the
     backlog and still drag onto today without the drop target scrolling off. */
  position:sticky; top:20px; align-self:flex-start;
  max-height:calc(100vh - 40px); overflow-y:auto;
  scrollbar-gutter:stable;
}

Three details make this work: align-self:flex-start (a stretched flex child can't stick — it's already full height), max-height + overflow-y:auto (the pinned pane scrolls internally), and scrollbar-gutter:stable (reserves scrollbar width so content doesn't jump sideways when the scrollbar appears). Note also that the comment itself is a design artifact: it records why, which is exactly what the next reader — or the next you — needs.

min-width:0, the flexbox tax

Both columns and several inner elements declare min-width:0. By default a flex child refuses to shrink below its content's intrinsic width, so one long unbreakable string (a URL, a branch name) blows the layout open. min-width:0 waives that, which is also the prerequisite for text-overflow ellipsis on session titles: white-space:nowrap; overflow:hidden; text-overflow:ellipsis only truncates once the element is actually allowed to be narrow.

Responsive: collapse, don't redesign

@media (max-width:880px){
  .grid{flex-direction:column;gap:44px}
  .board-col{position:static;max-height:none;overflow:visible; …}
  .resizer{display:none}
}

One breakpoint. Below it, the columns stack, the sticky behavior and the resizer are simply switched off. For a personal tool this is the right amount of responsive design: the page stays usable on a narrow window without maintaining a second layout. (Grid does appear where it fits better: Today's Focus is display:grid; grid-template-columns:1fr 1fr — two equal cells with no sizing negotiation is exactly what grid is for.)

Chapter 6

A component vocabulary in pure CSS

Five recurring components carry the whole interface. None needs JavaScript to look or behave the way it does.

6.1 · Pills

Small metadata badges: mono font, 11px, border-radius:999px (an arbitrarily large radius always yields a perfect capsule at any width). The base pill is neutral; variants only swap the color trio — accent-tinted for project names, purple for the Cowork source badge. Unicode glyphs stand in for an icon set: .pill.branch::before{content:"⎇ "}.

Pill variants

6.2 · Status dots — state as a parent class

A task's status is written once, as a class on the row (status-doing), and descendant selectors restyle the parts: 

.dot{width:10px;height:10px;border-radius:50%;border:1.5px solid var(--line-strong)}
.status-backlog .dot{border-color:var(--faint)}       /* empty circle = waiting */
.status-doing   .dot{background:var(--accent); …}     /* filled red = active   */
.status-done    .dot{background:var(--done); …}       /* filled gray = spent   */
.status-done .title{color:var(--done);text-decoration:line-through;
                    text-decoration-thickness:1px}

The encoding is shape-and-fill, not just hue: empty vs. filled reads even where the colors don't. When status changes, JS swaps one class and the dot, label, and strikethrough all follow — the CSS is the state machine's display layer. (The text-decoration-thickness:1px keeps the strikethrough from feeling like a censor bar at 15px text.)

6.3 · Hover-reveal actions, with the keyboard escape hatch

.row .del{ …; opacity:0; transition:opacity .12s }
li.row:hover .del, li.row:focus-within .del{ opacity:1 }

Destructive buttons stay invisible until the row is hovered, keeping thirty rows from showing thirty ✕s. The crucial half is :focus-within: tabbing into the row also reveals the button, so keyboard users aren't locked out by a mouse-only affordance. Any time you hide an action behind hover, pair it with :focus-within.

6.4 · The ledger rail — a timeline from two pseudo-elements

The daily log's vertical timeline uses no extra markup at all. The container draws a 2px line in its left padding via ::before; each entry draws a dot on that line via its own ::before — and here the ring trick from chapter 4 returns with a twist: 

.ledger{position:relative;padding-left:26px}
.ledger::before{content:"";position:absolute;left:5px;top:6px;bottom:6px;
  width:2px;background:var(--line-strong)}          /* the rail */
.agent::before{content:"";position:absolute;left:-26px;top:15px;
  width:10px;height:10px;border-radius:50%;background:var(--accent);
  box-shadow:0 0 0 3px var(--paper)}               /* the dot punches a hole */

The paper-colored ring around each dot visually erases the rail where the dot sits, so the line appears to pass behind it. No masking, no SVG — one box-shadow.

6.5 · Collapsible groups: native <details>, restyled

Each project group is a <details>/<summary> element, so expand/collapse, keyboard toggling, and open-state come free from the browser. CSS removes the default marker and substitutes a rotating caret: 

.agent > summary{list-style:none}                     /* drop default marker */
.agent > summary::-webkit-details-marker{display:none}
.agent > summary::before{content:"▸";transition:transform .15s}
.agent[open] > summary::before{transform:rotate(90deg)}

Chapter 7

State, feedback, and motion

Focus is the accent's most important job

.add input:focus{
  outline:2px solid var(--accent); outline-offset:1px;
  border-color:transparent;
}

Every focusable element gets the same treatment: a 2px accent outline, offset so it doesn't crowd the content, with the regular border hidden so the two don't stack into a muddy double edge. Because it's one consistent signature, focus reads as "the system is listening here" everywhere — inputs, editable titles, the date picker. Never remove focus outlines; restyle them once, in the accent, and reuse everywhere.

Click or tab into this input

Motion: three animations, all with jobs

  • Rise — log entries fade in and travel 4px upward over 0.35s when a day loads. Just enough to say "this content arrived", far below the threshold of annoyance on the hundredth viewing.
  • Spin — the ↻ regenerate button rotates while its request is in flight, keyed to :disabled. The busy indicator is the button; no spinner element is added or removed.
  • Micro-transitions — hovers and reveals run at 0.1–0.15s. At that duration a transition is felt as smoothness rather than watched as movement.
.agent{animation:rise .35s ease both}
@media (prefers-reduced-motion:reduce){.agent{animation:none}}
@keyframes rise{from{opacity:0;transform:translateY(4px)}to{opacity:1;transform:none}}

The reduced-motion guard is one line and non-negotiable: users who disable animation OS-wide (often for vestibular reasons) get the content instantly.

Ambient feedback: the save note

Saving is silent and debounced (chapter 8), so the only confirmation is a small fixed toast in the corner — mono, muted, fading in for 900ms. It has a second, longer-lived amber variant (.savenote.stale) for the one message that matters: "board changed elsewhere — reloaded, redo your last edit." Routine feedback whispers; exceptional feedback changes color and sticks around. Amber is also used for the missing-sessions-folder warning — a third semantic color, spent only on "something needs your attention but isn't broken."

Chapter 8

JS patterns that serve the design

The JavaScript isn't the subject of this guide, but four of its patterns are really front-end design decisions wearing JS clothes.

8.1 · A three-line "framework"

const $  = s => document.querySelector(s);
const el = (t, c) => { const e = document.createElement(t); if (c) e.className = c; return e; };
const esc = s => String(s).replace(/[&<>"]/g, m => ({'&':'&amp;', …}[m]));

Everything renders through el(): build nodes, set textContent, append. Using textContent instead of innerHTML for user data makes cross-site-scripting impossible by construction; esc() exists for the few places markup is assembled as a string. The broader pattern is state → render: the board lives in one JS object, and renderTasks() rebuilds its list from that object after every change. That's the same one-way mental model as React, minus React — the DOM is never the source of truth. At a few dozen rows, "throw it away and rebuild" costs nothing and eliminates the whole category of the-DOM-and-the-data-disagree bugs.

8.2 · Edit-in-place with contenteditable

Task titles are just <div contenteditable> — no hidden input swapped in on click, no edit mode. Blur commits (empty edits revert), Enter is remapped to blur, and the placeholder for the empty day-summary is pure CSS: 

.day-summary:empty::before{
  content:attr(data-placeholder); color:var(--faint); font-style:italic;
}

The :empty::before + attr() trick gives any editable element input-style placeholder text with zero JS.

8.3 · Filtering by hiding, not re-rendering

The search box hides non-matching rows with a class (li.row.filtered-out{display:none}) rather than re-rendering a filtered list. The source comment explains why: the rows are drag-and-drop sources whose identity and indices must stay stable — re-rendering a subset would break dragging while a filter is active. A general lesson: when elements carry interaction state, prefer hiding to rebuilding.

8.4 · Optimistic saves with a stale-tab guard

Every mutation renders immediately, then a debounced save fires 350ms after the last change — type three characters, get one network call. The interesting part is what happens when a save is wrong: the board carries a revision counter, and a POST from a tab holding an outdated copy gets a 409, at which point the client adopts the server's board and tells the user, rather than silently overwriting newer data. That protocol was added after a real incident (a long-lived stale tab cost days of task history). UI honesty — the amber toast, the resync-on-focus — is the user-visible half of a data-integrity design.

8.5 · One more: the inline SVG icon

The search field's magnifying glass is an SVG embedded directly in the CSS as a data: URI background image — one icon, zero requests, zero icon fonts. Worth knowing for any "I need exactly one icon" moment. (Its stroke color is hard-coded in the URI, which is the technique's one wart: data-URI SVGs can't see CSS variables. Modern alternative if you ever need themed icons: CSS mask-image with background-color:var(--faint).)

Chapter 9

The decision ledger: alternatives considered and discarded

Design maturity shows less in what a project has than in what it deliberately doesn't. These are from the project's recorded history — real decisions, not reconstructions — except where marked inferred.

ChosenDiscardedWhy
Vanilla JS + stdlib server Framework, bundler, npm Founding constraint: nothing to install, nothing to break, one file to read. Every problem so far has fit inside it.
Rose paper / brick accent Default light gray-on-white Explicit aesthetic preference, recorded as such. A personal tool should look like its person.
Light theme only Dark mode Never requested; the paper look is the identity. The token architecture means adding one later is ~20 lines (this guide's own dark mode is exactly that, as proof).
Rev counter + 409 on stale saves Last-write-wins autosave Last-write-wins was the original design — until a stale background tab silently clobbered days of task history. Losing one edit loudly beats losing days silently.
Daily board backups Weekly cadence Considered and rejected: daily bounds worst-case loss at one day, and the file is ~20KB — the storage argument for weekly didn't exist.
Read-only rows show true status Styling all past-day items as done The original hardcoded done-styling made in-progress work look finished — a display shortcut that misled the owner and compounded into apparent data loss. Visual state must never contradict data state.
Filter by hiding rows Re-rendering a filtered list Rows are drag sources; rebuilding them mid-filter breaks drag identity and indices (documented in a source comment).
Sticky board with own scroll rail Both columns scroll with the page The drag-onto-today interaction dies if the drop target can scroll off-screen (documented in a source comment).
Native <details> disclosure JS accordion (inferred) The browser already ships toggle behavior, keyboard support, and open-state; CSS restyles the marker. Zero JS to maintain.
One breakpoint, columns stack A designed mobile experience Mobile came up as "tell me what it'd take" — an informational question, deliberately not built. Matching effort to actual need is a design decision too.
Show sessions ≥ 1,000 tokens The original 5,000 cutoff Thresholds are design: 5,000 kept the ledger tidy but hid real work. Tuned on feedback, not theory.

The meta-lesson across the ledger: almost every reversal came from watching the thing get used — a scare, a misleading display, a hidden session — not from a redesign. Small tools earn their design through incidents.

Chapter 10

Exercises

Each of these is small, touches the real file, and exercises one chapter. The token architecture means none of them should require touching more than a handful of lines.

  1. Re-skin it (ch. 1–2). Change only the :root block to a different palette — say, a cool slate paper with a pine-green accent. Keep the ramp structure: ground, surface, three inks, two lines, one accent in three strengths. If you have to edit anything outside :root, you've found a hard-coded color that escaped the token system (there are a couple — the Cowork purple and some pill border colors are honest examples of token drift).
  2. Add dark mode (ch. 1, 9). Wrap a second set of token values in @media (prefers-color-scheme: dark). Don't invert — darken the paper toward warm brown-black and lighten the accent (dark backgrounds need brighter accents for the same perceived contrast; compare this page's two themes).
  3. Add a fourth status (ch. 6). Introduce blocked: extend the STATUSES array and add a .status-blocked ruleset. Notice how little JS changes — the CSS state-class pattern absorbs the feature.
  4. Build a pill variant (ch. 6). Add .pill.stale using the amber warn colors for sessions older than some age. One selector, three declarations.
  5. Break min-width:0 on purpose (ch. 5). Delete it from .log-col, add a session with a very long unbroken title, and watch the layout blow out. The fastest way to internalize the flexbox minimum-size rule is to see the failure once.
